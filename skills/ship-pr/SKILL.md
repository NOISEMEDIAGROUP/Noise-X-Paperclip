# Ship PR (Paperclip Agent)

You are running an automated ship workflow within a Paperclip heartbeat. Take the current
branch from working state to a merged-ready PR.

## When to use

Use this skill:
- When your implementation work is complete and you need to ship
- When a task status indicates "ready to ship"
- After completing code review fixes

## Step 1: Pre-flight

```bash
_BRANCH=$(git branch --show-current)
_BASE=$(gh pr view --json baseRefName -q .baseRefName 2>/dev/null || gh repo view --json defaultBranchRef -q .defaultBranchRef.name 2>/dev/null || echo "main")
echo "BRANCH: $_BRANCH | BASE: $_BASE"
```

If on the base branch, **abort**: "Cannot ship from the base branch. Work on a feature branch."

```bash
git status
git diff "$_BASE"...HEAD --stat
git log "$_BASE"..HEAD --oneline
```

## Step 2: Sync with base branch

```bash
git fetch origin "$_BASE" && git merge "origin/$_BASE" --no-edit
```

If merge conflicts occur:
- Try auto-resolving simple conflicts (VERSION, CHANGELOG ordering, lockfiles)
- If conflicts are complex, **stop** and report them as a Paperclip comment

## Step 3: Run tests

Run the project's test suite. Detect the test runner:

```bash
# Check what test infrastructure exists
[ -f package.json ] && grep -q '"test"' package.json && echo "HAS_NPM_TEST=true"
[ -f Makefile ] && grep -q '^test:' Makefile && echo "HAS_MAKE_TEST=true"
[ -f Cargo.toml ] && echo "HAS_CARGO_TEST=true"
```

Run the appropriate test command. If tests fail, **stop** and report failures.

## Step 4: Code review

Run the `code-review` skill logic (Pass 1 CRITICAL + Pass 2 INFORMATIONAL) against the diff.

- If CRITICAL issues found: fix them, commit the fixes, and re-run tests
- If only INFORMATIONAL issues: note them for the PR body

## Step 5: Create commits

Create small, logical commits that work well with `git bisect`:

1. **Analyze the diff** and group changes into logical commits
2. **Ordering** (earlier first):
   - Infrastructure: config, migrations, route additions
   - Core: models, services, libraries (with their tests)
   - Integration: controllers, views, components (with their tests)
   - Housekeeping: VERSION, CHANGELOG, docs (final commit)
3. **Rules:**
   - A module and its test file go in the same commit
   - Each commit must be independently valid (no broken imports)
   - If total diff is small (<50 lines, <4 files), a single commit is fine
4. **Commit message format:**
   ```
   <type>: <summary>

   Co-Authored-By: <agent-name> (Paperclip) <noreply@paperclipai.com>
   ```
   Types: feat, fix, chore, refactor, docs, test

## Step 6: Push

```bash
git push -u origin "$_BRANCH"
```

## Step 7: Create PR

```bash
gh pr create --base "$_BASE" --title "<type>: <summary>" --body "$(cat <<'EOF'
## Summary
<bullet points describing what changed and why>

## Code Review
<findings from Step 4, or "No issues found.">

## Test Results
- [x] All tests pass

## Paperclip Context
- Task: $PAPERCLIP_TASK_ID
- Agent: $PAPERCLIP_AGENT_ID
- Run: $PAPERCLIP_RUN_ID

Generated by Paperclip agent
EOF
)"
```

Report the PR URL back via Paperclip comment on the task.

## Rules

- **Never skip tests.** If tests fail, stop and report.
- **Never force push.** Use regular `git push` only.
- **Never ship from the base branch.**
- **Always include task context** in the PR body.
- **Split commits for bisectability** — each commit = one logical change.
- Update the Paperclip task status to reflect shipping progress.
